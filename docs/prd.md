# 《项目代号：别点错》小程序游戏 PRD v0.1

## 1. 项目概述

### 1.1 项目背景
市面上大量小游戏要么偏重数值成长，要么偏重复杂操作，但用户在碎片时间里，很多时候想要的是一种更轻、更直接的情绪体验：

- 不需要学习成本
- 一上手就能玩
- 每关只花几秒到十几秒
- 有一种“离谱但好笑”的爽感
- 失败不挫败，反而想立刻再试一次

本项目希望做一个**轻量级关卡制解压小游戏小程序**，通过大量短平快、无厘头、反套路的小关卡，为用户提供一种“被游戏耍了一下，但又很好笑”的体验。

### 1.2 项目定位
一个以**无厘头 / 解压 / 反套路 / 极简操作**为核心体验的关卡制小游戏合集。

### 1.3 核心一句话定义
**这是一个让玩家用最简单的操作，体验最离谱通关方式的无厘头小游戏。**

---

## 2. 产品目标

### 2.1 用户目标
让用户在碎片时间里获得：

- 轻松感
- 解压感
- 好笑感
- 想继续点下一关的冲动

### 2.2 商业/产品目标（第一阶段）
第一版先不追求复杂商业化，优先验证三个事情：

1. 用户是否愿意连续玩多关
2. 用户是否愿意分享“某一关太离谱了”
3. 这种内容形式是否适合长期扩展为小游戏合集

### 2.3 第一阶段成功标准
首版上线后，重点观察：

- 平均单次游玩关卡数
- 单关失败后重试率
- 第 1 关到第 10 关的留存漏斗
- 用户主动分享/截图吐槽的频次
- 哪类“无厘头关卡”最容易传播

## 3. 目标用户

### 3.1 核心用户
- 18-35 岁轻度游戏用户
- 短视频/梗文化/网络热梗接受度高的人群
- 喜欢“离谱”“抽象”“逆天”“玩梗”内容的人
- 通勤、排队、摸鱼时想随手点几下的人

### 3.2 用户画像

#### 用户A：碎片时间解压型
- 上班摸鱼、地铁通勤
- 不想动脑太多
- 希望几秒一个反馈
- 喜欢“这也行？”的感觉

#### 用户B：梗图传播型
- 玩到某一关会截图发朋友
- 会说“这游戏有病但很好笑”
- 更容易带来裂变传播

#### 用户C：关卡征服型
- 想看看后面还有多离谱
- 会因为“再来一关”持续玩下去
- 对关卡创意敏感

## 4. 产品核心体验

### 4.1 核心体验关键词
- 简单
- 荒诞
- 反常识
- 快反馈
- 低失败成本
- 高情绪价值

### 4.2 体验原则

#### 原则1：操作必须简单
每一关尽量只使用这些交互：

- 点击
- 长按
- 拖拽
- 滑动
- 多次点击
- 设备轻微交互（后续可扩展，如摇一摇）

#### 原则2：答案不能太正常
玩家第一反应想到的做法，最好不是正确答案。

#### 原则3：失败要有趣
失败不是惩罚，而是包袱的一部分。

#### 原则4：单关节奏要极短
- 理想通关时间：3~15 秒
- 稍复杂关卡：20~30 秒
- 不做长流程单关

#### 原则5：通关反馈要爽
过关时必须立刻给到：

- 明显视觉反馈
- 轻快音效
- 夸张/搞笑文案
- “下一关”强引导

## 5. 产品形式

### 5.1 产品结构
第一版采用：

- 单主线关卡模式
- 线性解锁
- 每关独立场景
- 每关一个核心笑点 / 包袱点 / 反转点

### 5.2 内容组织方式
方案：

- 第 1~10 关：建立用户认知，偏简单、偏离谱
- 第 11~30 关：加入误导和轻度观察
- 第 31~50 关：加入组合操作和连续误导

第一版 PRD 建议先做 **30 关 MVP**。

原因：

- 足够验证玩法
- 开发量可控
- 也能初步观察用户疲劳点

## 6. 核心玩法设计

### 6.1 基础玩法循环
玩家进入关卡后：

1. 看到一个很简单的目标 / 提示
2. 按常规思路尝试
3. 失败或发现不对劲
4. 观察画面细节 / UI / 文案
5. 用离谱方式触发正确解
6. 通关
7. 快速进入下一关

### 6.2 核心乐趣来源

#### 乐趣1：预期被打破
比如提示写“点击开始”，真正能过关的是点角落里不起眼的“结束”。

#### 乐趣2：系统被拿来玩梗
比如按钮、标题、设置、返回、弹窗本身都可能成为谜题的一部分。

#### 乐趣3：失败本身有笑点
玩家点错时出现吐槽文案，而不是冷冰冰的“失败”。

#### 乐趣4：超级短反馈
几秒内给出结果，减少流失。

## 9. 首版关卡设计方向

下面先给出一批适合 MVP 的关卡方向，不是最终全量，但足够作为第一版内容基底。

### 9.1 新手关（建立产品调性）

#### 第1关：通关
**画面：**
- 中间一个大按钮，写着“通关”

**玩法：**
- 直接点击即可通关

**目的：**
- 先让玩家建立“就这？”的轻松预期
- 给一个假安全感

#### 第2关：不要点
**画面：**
- 一个大按钮写“不要点”

**玩法：**
- 点击它即可通关

**目的：**
- 立刻建立“提示不可信”的规则

#### 第3关：再点一下
**画面：**
- 一个按钮写“点击 1 次通关”

**玩法：**
- 连点 2 次才过

**目的：**
- 建立“字面意思会骗人”

#### 第4关：按钮不见了
**画面：**
- 文案：“找到通关按钮”
- 屏幕上没有按钮

**玩法：**
- 玩家点击文案里的“通关”二字过关

**目的：**
- 引导用户开始观察文字本身

#### 第5关：别急
**画面：**
- 一个倒计时 3 秒
- 文案：“快点击！！！”

**玩法：**
- 什么都不做，等倒计时结束通关

**目的：**
- 第一次引入“什么都不做也是答案”

### 9.2 轻误导关

#### 第6关：拖动标题
- 需要把顶部标题拖开，露出隐藏按钮

#### 第7关：长按
- 点没用，长按 2 秒才触发通关

#### 第8关：按钮会跑
- 点按钮时按钮乱跑，真正通关方式是点背景

#### 第9关：假失败
- 点击后出现“失败了”
- 再点一次“失败了”弹窗上的表情，反而通关

#### 第10关：滑掉它
- 一个看似正常按钮点不动
- 需要把按钮文字滑走，露出真正答案

### 9.3 抽象搞笑关

#### 第11关：太亮了
- 太阳很刺眼
- 用手指把太阳拖到屏幕外即可过关

#### 第12关：让我安静
- 屏幕上一个角色一直说话
- 把它嘴巴拖走 / 用按钮堵住嘴

#### 第13关：不要站着
- 一个角色站在那
- 玩家拖动他坐下，通关

#### 第14关：开门
- 门打不开
- 其实是把“开门”两个字拖到门上

#### 第15关：请低调一点
- 一个特别夸张的“过关”按钮
- 真正答案是把按钮缩小到很小才通关

### 9.4 利用UI做文章的关卡

#### 第16关：关闭广告
- 屏幕中间一个假广告弹窗
- 关闭按钮是假的
- 真正要点的是弹窗标题栏的一个小叉

#### 第17关：设置一下
- 进入“设置”页面，把某个开关关掉才能通关

#### 第18关：返回上一页
- 不是系统返回
- 而是点页面里手绘的“返回”箭头

#### 第19关：加载中
- 转圈一直不结束
- 把“加载中”文字拖走后过关

#### 第20关：确认
- 有两个按钮“确认”“取消”
- 点“取消”才过关

## 10. 系统功能设计

### 10.1 关卡系统

#### 功能说明
- 默认从第 1 关开始
- 通关后解锁下一关
- 支持继续上次进度
- 支持从已解锁关卡进入

#### 数据字段建议
- `levelId`
- `status`（locked / unlocked / passed）
- `bestTime`
- `attemptCount`
- `passTimestamp`

### 10.2 提示系统（可选）

#### 目标
避免玩家卡太久导致流失。

#### 方案建议
- 单关失败 3 次后可显示“要不要一点提示？”
- 提示分 2 层：
  - 弱提示：注意看看文字
  - 强提示：试试拖动顶部区域

#### 原则
提示不能直接把答案全说完，否则笑点就没了。

### 10.3 失败反馈系统

#### 目标
增强情绪感，不让失败变成负担。

#### 示例文案
- 你也太老实了吧
- 谁让你真按提示来的
- 这游戏不讲武德
- 再想想，画面上还有东西没被你怀疑
- 你看起来很像个正常人，但这关不需要正常人

### 10.4 通关反馈系统

#### 示例文案
- 就这？你还真过了
- 离谱，但合理
- 恭喜，你已经开始不正常了
- 这一关设计师先笑了
- 继续，后面更抽象

## 11. 数值与节奏设计

### 11.1 单关时长目标
- 新手关：3~8 秒
- 普通关：8~20 秒
- 稍复杂关：20~30 秒

### 11.2 难度曲线
整体采用锯齿式增长：

- 前几关必须爽快
- 中段开始加入误导
- 每 4~5 关给一个轻松梗关缓冲
- 不让用户持续高压思考

### 11.3 失败容忍
- 失败后 1 秒内可重试
- 不做复杂惩罚
- 不扣体力
- 不扣生命

## 12. 美术风格建议

### 12.1 风格关键词
- 极简
- 夸张
- 脑洞
- 手绘感 / 轻卡通
- 梗图式表达

### 12.2 美术方向建议

#### 方向A：极简 UI 风
- 白底 + 黑线 + 高饱和重点按钮
- 成本低，适合快速做关卡

#### 方向B：轻手绘梗图风
- 更有个性
- 更利于传播
- 但制作成本更高

#### 第一版建议
先用 **极简 UI 风 + 少量夸张表情元素**，优先把玩法跑通。

## 13. 音效与动效建议

### 13.1 音效
- 点击音效：轻快清脆
- 失败音效：滑稽/短促
- 通关音效：爽快、有“梗感”
- 特殊关卡：可加入夸张拟声

### 13.2 动效
- 按钮点击弹性反馈
- 通关时屏幕轻微抖动
- 错误操作时元素抖动/翻白眼/闪一下
- 下一关切换要快，不拖泥带水

## 14. 技术方案建议

### 14.1 前端技术栈
- Vue 3
- Vite
- TypeScript
- Phaser

### 14.2 前端职责划分建议

#### Vue 负责
- 页面结构
- 首页/设置/关卡选择等 UI
- 状态管理
- 路由
- 数据上报
- 与后端通信

#### Phaser 负责
- 关卡内交互
- 游戏对象管理
- 动画
- 碰撞/点击/拖拽逻辑
- 关卡脚本执行

### 14.3 架构建议
建议采用 **“壳层 + 游戏内核”** 模式：

- Vue 负责外层应用壳
- Phaser 作为关卡运行容器
- 每一关抽象成独立 `level config / level script`

#### 关卡配置建议
每一关尽量数据驱动：

- 基础文案
- 元素布局
- 可交互对象
- 通关条件
- 失败条件
- 提示文案
- 特殊行为脚本

这样后面扩关会快很多。

## 17. 风险点

### 17.1 最大风险

#### 风险1：创意消耗太快
这种游戏最值钱的是关卡脑洞，不是技术本身。

#### 风险2：玩家觉得“无聊而不是搞笑”
无厘头和无聊只有一线之隔。

#### 风险3：后期内容生产压力大
必须有一套可以持续产关的机制。

---

## 18. 解决思路

### 18.1 内容模板化
把关卡创意拆成几种模板：

- 文字误导型
- UI 欺骗型
- 拖拽反转型
- 等待型
- 长按型
- 假按钮型
- 系统元素玩梗型

以后新关卡可以按模板批量生产。

### 18.2 建立关卡创意库
每次想到一句梗、一个反转、一个离谱操作，就记进库里，再后续加工成关卡。

### 18.3 先做少量精品
第一版不要贪多，先做 20~30 个真有趣的关卡。

## 21. 一句话总结

这是一个靠 **极简操作 + 反套路脑洞 + 快速情绪反馈** 驱动的无厘头关卡小游戏项目。  
第一版最重要的不是做复杂系统，而是验证：

**“用户会不会因为这些离谱的小关卡，连续玩下去，并且觉得好笑、想分享。”**
